# AI 长任务通用工作流（Phase-Contract Workflow）

> 一份可复用、可迁移到不同业务需求的 AI 驱动长任务执行方法论。
> 源自 Aegis 项目实践的抽象与提炼，剥离业务后可直接套用到任何领域。

---

## 0. 它解决的根本问题

AI 在连续工作 3 小时以上的任务里会稳定出现四类失败模式：

1. **进度漂移**：记不清做到哪一步，或把"打算做"当成"已经做完"
2. **边界越界**：改着改着顺手动了无关模块，引入回归
3. **目标遗忘**：长时间后忘记最初约束（版本、依赖禁区、质量底线）
4. **压缩失忆**：上下文一旦被压缩或换会话，原本的隐式状态全部丢失

传统做法（写一份大 Plan.md、让 AI 边做边自己判断）在 2–3 小时后必然崩。

本工作流的核心主张是：

> **把长任务里"需要稳定性"的部分从 AI 记忆里剥离，外部化为仓库里的可执行合同。AI 只做"在小窗口内执行小合同"这一件事。**

---

## 1. 核心思想：Phase-Contract Model

把任意长任务建模为一条**有序的合同链**：

```
合同链 = [ Phase₀ → Phase₁ → Phase₂ → ... → Phaseₙ ]

每个 Phase = ⟨ 定位合同, 执行合同 ⟩
每次执行 = ⟨ 全局约束 + 定位合同 + 执行合同 ⟩ → 交付物 → 写回状态
```

三个不变量撑起整套方法论：

| 不变量 | 含义 | 违反后果 |
|--------|------|---------|
| **I1 单一活跃合同** | 任意时刻只有一个 phase 处于"执行中" | 目标漂移、边界越界 |
| **I2 三文件上下文** | 任意时刻工作窗口只装 3 份文档 | 注意力被稀释，长任务崩溃 |
| **I3 完成即事实** | phase 完成必须写入外部状态文件 | 进度失真、无法恢复 |

后续所有设计都是为了让这三个不变量**无法被违反**，而不是"希望 AI 遵守"。

## 2. 八原则（可打印挂墙）

| # | 原则 | 一句话 | 落地动作 |
|---|------|--------|---------|
| P1 | **状态外部化** | 进度写到文件，不写到记忆 | 设一份 `state` 文件，完成必须写入 |
| P2 | **调度与执行分离** | 脚本决定做什么，AI 决定怎么做 | 用 `planctl` 类脚本承担调度 |
| P3 | **三文件上下文律** | 全局约束 + 阶段定位 + 本次边界 | manifest 强制声明 `required_context` |
| P4 | **双层合同** | "是什么"和"能碰什么"分开写 | `phases/*` 与 `execution/*` 两套文档 |
| P5 | **依赖强制校验** | 顺序由脚本守护，不信任直觉 | `depends_on` + `--strict` 依赖检查 |
| P6 | **完成即写入** | 完成 = 写 state + 刷 handoff | 两个动作缺一不继续 |
| P7 | **固定恢复协议** | 压缩后恢复永远三步：manifest → handoff → next | 写进仓库级 AI 指令 |
| P8 | **里程碑外部化到 git 远端** | 每个 phase 完成即 commit + push，形成可回滚的远端记录 | `complete` 自动 `git add -A / commit -F - / push`，地道英文 commit message，失败只 warn 不回滚 state |

## 3. 制品体系（Artifact Layout）

> **基础设施前提**：目标项目根目录必须是 git 工作区（`git rev-parse --is-inside-work-tree` 返回 `true`）。Phase-Contract 的路径白名单比对、phase 级回滚、`complete` 时的 diff 审计都以 git 为客观账本；缺少 git 则 `execution/*` 的“允许改动 / 禁止改动”无法被核验，Quality Gate 悬空。唯一的豁免通道是显式设置 `PHASE_CONTRACT_ALLOW_NON_GIT=1` 并在 `common.md` 中以风险段记录偏离与补偿方案，默认路径禁止豁免。

一个完整实施需要且仅需要以下 7 类文件。命名可改，职责不能合并。

```
<project>/
├── plan/
│   ├── manifest.yaml            # ① 流程清单：顺序 + 依赖 + required_context
│   ├── common.md                # ② 全局硬约束（长期不变项）
│   ├── phases/phase-X.md        # ③ 定位合同：是什么 / 目标 / 完成判定
│   ├── execution/phase-X.md     # ④ 执行合同：允许改 / 禁止改 / 交付检查
│   ├── state.yaml               # ⑤ 执行账本（脚本写入，勿手改）
│   ├── handoff.md               # ⑥ 压缩恢复锚点（脚本写入，勿手改）
│   └── workflow.md              # （说明文档，可选）
├── scripts/
│   └── planctl                  # ⑦ 唯一流程入口
└── .github/
    └── copilot-instructions.md  # ⑧' 仓库级 AI 硬约束（Copilot）
├── CLAUDE.md                    # ⑧'' 同上（Claude Code，内容一致）
└── AGENTS.md                    # ⑧''' 同上（Codex / 通用 agent，内容一致）
```

### 各制品职责详解

| 制品 | 回答的问题 | 变更频率 | 谁来写 |
|------|-----------|----------|--------|
| manifest | 有哪些 phase？什么顺序？每个 phase 要读哪些文档？ | 启动时写一次，极少改 | 人类 |
| common | 整个项目永远不允许违反的规则是什么？ | 启动时写一次，极少改 | 人类 |
| phases/phase-X | 这个阶段要达到什么状态？怎么算完成？ | 进入该 phase 前写 | 人类 / AI 辅助 |
| execution/phase-X | 这次实施可以碰什么、不能碰什么、怎么验收？ | 进入该 phase 前写 | 人类 / AI 辅助 |
| state | 到目前为止完成了什么？ | 每个 phase 完成时写 | **脚本** |
| handoff | 压缩后要怎么继续？下一步读什么？ | 每个 phase 完成时写 | **脚本** |
| planctl | — | 启动时实现一次 | 人类（可直接复用现成实现） |
| copilot-instructions / CLAUDE / AGENTS | AI 开会话时要先做什么？（三份同步） | 启动时写一次 | 人类 |

**关键设计**：state 和 handoff 由脚本写、禁止人手改。这让"进度"成为客观事实而非叙述。

## 4. 双层合同模板

### 4.1 定位合同 `phases/phase-X.md`

```markdown
# Phase X: <标题>

## 阶段定位
<这个阶段在整个链条中解决什么问题，一句话>

## 必带上下文
- plan/common.md
- Phase X-1 已完成

## 阶段目标
- <可验证的目标 1>
- <可验证的目标 2>

## 实施范围
- <涉及的模块/文件/能力>

## 本阶段产出
- <具体交付物>

## 明确不做
- <最容易被误伤的越界项>

## 完成判定
- <机器可检或逐项可勾的客观检查>

## 依赖关系
- 依赖 Phase X-1
```

**写作铁律**：
- "阶段目标"必须可验证；写不出验证方式的目标，是无效目标
- "明确不做"通常比"阶段目标"更重要——它保护边界
- "完成判定"禁止出现"良好"、"合理"、"基本完成"等主观词

### 4.2 执行合同 `execution/phase-X.md`

```markdown
# Phase X 执行包

## 必带上下文
- plan/common.md
- plan/phases/phase-X.md

## 执行目标
- <本次实施要解决的具体问题>

## 本次允许改动
- <路径/模块 1>
- <路径/模块 2>

## 本次不要做
- <禁止涉及的文件/重构/feature>

## 交付检查
- <逐项可勾的客观清单>

## 执行裁决规则
- 如果 <触发条件>，直接判定无效
- 如果引入超出本 phase 的范围漂移，直接回退到规划边界
```

**写作铁律**：
- execution 是**围栏**，不是**脚手架**。不要写"第一步做 A、第二步做 B"——那是在替 AI 思考，AI 反而会机械照做并越界
- "允许改动"必须是一份**白名单**（路径/模块级），不是"只要和本阶段相关就行"
- "裁决规则"是一票否决机制：一旦触发，无论工作量多大都要回退

## 5. 调度脚本契约（planctl 抽象接口）

脚本实现可用任意语言，但必须提供以下 5 条命令。Aegis 的 Ruby 实现可直接复用。

| 命令 | 输入 | 输出 | 作用 |
|------|------|------|------|
| `resolve <phase-id>` | phase id | 该 phase 的 required_context + 依赖检查结果 | 单 phase 启动 |
| `next` | — | 下一个未完成且依赖满足的 phase | 连续推进 |
| `status` | — | 已完成 / 可执行 / 被阻塞三组清单 | 总览 |
| `complete <phase-id>` | summary + next_focus | 写入 state.yaml | 标记完成 |
| `handoff --write` | — | 刷新 handoff.md | 刷新恢复锚点 |

**两个硬要求**：

1. `resolve` / `next` 必须支持 `--strict`：依赖不满足时以非零退出码终止，禁止返回"可以开始"的假信号
2. `complete` 必须在写入前重新校验 `depends_on`，不信任调用方传入的 phase id

## 6. 标准执行循环（Golden Loop）

```
┌─ [0] 开局 ─────────────────────────────────────────────┐
│  读 manifest + handoff                                 │
│  跑  planctl next --strict                             │
└────────────────────────┬───────────────────────────────┘
                         ↓
┌─ [1] 装载 ─────────────────────────────────────────────┐
│  只读 required_context 里的 3 份文档                   │
│  禁止扩读其他 phase 文档                               │
└────────────────────────┬───────────────────────────────┘
                         ↓
┌─ [2] 实施 ─────────────────────────────────────────────┐
│  严格遵守 execution 的 允许 / 禁止 / 裁决规则          │
│  越界立即停手回到规划边界                              │
└────────────────────────┬───────────────────────────────┘
                         ↓
┌─ [3] 自检 ─────────────────────────────────────────────┐
│  逐项过"完成判定"和"交付检查"                          │
│  任一项不过 → 不算完成，不得进入 [4]                   │
└────────────────────────┬───────────────────────────────┘
                         ↓
┌─ [4] 写回 ─────────────────────────────────────────────┐
│  planctl complete <id> --summary ... --next-focus ...  │
│  事实落盘                                              │
└────────────────────────┬───────────────────────────────┘
                         ↓
┌─ [5] 刷新锚点 ─────────────────────────────────────────┐
│  planctl handoff --write                               │
│  恢复锚点就位                                          │
└────────────────────────┬───────────────────────────────┘
                         ↓
           还有下一 phase？  ── 否 → 整体任务结束
                         │
                         是
                         ↓
                     回到 [0]
```

**压缩/新会话中断点**：任何时刻被压缩，**从 [0] 重新插入即可无损续跑**。这是连续工作 5+ 小时的机械保障。

## 7. 切 phase 的方法论

切得好不好，直接决定这套工作流跑得动跑不动。

### 7.1 切 phase 的三个判据

1. **可独立交付**：这个 phase 做完后单独产出一份可验收的东西
2. **可客观验证**：完成判定能写成可勾选清单，不靠主观判断
3. **尺寸适中**：AI 单段实施窗口 30–90 分钟，总 phase 数 5–12 个

写不出客观完成判定，就是切错了。常见征兆：

| 征兆 | 原因 | 修正 |
|------|------|------|
| 完成判定只能写"功能跑起来了" | phase 覆盖了多个验收点 | 继续拆分 |
| 两个 phase 总在互相改同一批文件 | 切分维度错（应按模块，切成了按能力） | 换维度重切 |
| 某 phase 文档写不到 200 字 | 颗粒太细，应合并 | 合并 |
| 总 phase 超过 15 个 | 过度细分 | 引入 epic 层 |

### 7.2 切分常见维度

选一个主维度贯穿整个项目，不要混用：

- **按模块**：适合新产品从 0 到 1（shared → service → UI → release）
- **按层次**：适合重构（数据层 → 领域层 → 应用层 → 接口层）
- **按章节**：适合内容工程（骨架 → 第 1 章 → 第 2 章 → 终审）
- **按控制项**：适合合规整改（访问控制 → 日志审计 → 加密 → 应急）
- **按迁移波次**：适合框架升级（工具链 → 共享层 → 模块 A → 模块 B → 切流）

### 7.3 依赖设计

依赖（`depends_on`）是硬阀门，不是建议。三条规则：

1. **只写真依赖**：X 依赖 Y 当且仅当 X 的交付检查需要 Y 的产出。不要把"逻辑相关"写成依赖
2. **禁止循环**：任何循环依赖都是切错了，必须重构 phase 边界
3. **宁多不少**：漏写依赖比多写依赖危险得多，前者会让 AI 跑通却实际错了

## 8. 全局约束（common.md）撰写指南

### 该放什么

- 技术栈版本边界（最低/最高版本、禁用的 API）
- 依赖禁区（禁用的库、禁用的运行时）
- 质量底线（测试覆盖、签名、日志、性能）
- 非目标（明确不做的能力，用来对抗 scope creep）
- 视觉/交互/文案底线（如果是产品）
- 数据/隐私边界（如果涉及合规）

### 不该放什么

- 某个 phase 才关心的规则（放进那个 phase 的 execution）
- 会频繁变化的规则（放哪都不对，说明它不是"约束"而是"策略"）
- 实施步骤（这是 execution 的职责）

### 判据

> 一条规则是否该写进 common.md？问自己：**任意一个未来的 phase 都有可能违反这条规则吗？** 是，才写。

## 9. 仓库级 AI 指令（enforcement layer）

`.github/copilot-instructions.md`、`CLAUDE.md`、`AGENTS.md` 这三份内容一致的文件（分别服务于 GitHub Copilot、Claude Code、Codex/通用 agent）是**流程自动运转的最后一块拼图**。它们利用各家 AI 工具链在每次会话开头注入的主提示机制，把规约变成**执行前置条件**而不是事后检查。三份必须同步维护，任一份漂移都会破坏强制层。

至少需要声明以下 9 条硬约束：

1. 开始前必须先读 manifest，再跑 `planctl next --strict` 或 `resolve --strict`
2. 不得绕过脚本手工选择当前 phase
3. 不得跳过 `depends_on` 检查
4. 装载上下文必须严格按 `required_context` 顺序，不得扩读未来 phase 的文档
5. 长流程中工作窗口必须保持"只装 3 份文档"
6. 未运行 `complete` 的 phase 不视为完成
7. 每完成一个 phase 必须立即 `handoff --write`，再进入下一 phase
8. 压缩或新会话恢复只走"manifest → handoff → next"三步
9. 业务 phase 期间不得自行 `git commit` / `git push` / `git tag`；里程碑提交权统一交给 `planctl complete` 自动行使

**没有这一层，前面所有设计都依赖 AI 自觉。有了这一层，规约变成强制。**

## 10. 把它迁移到新项目的 30 分钟路径

1. **复制基础设施**（5 分钟）：把现成的 `planctl` 脚本、`workflow.md` 和 agent 指令模板（一份内容分别保存为 `.github/copilot-instructions.md` / `CLAUDE.md` / `AGENTS.md`）拷到新项目，改项目名
2. **写 manifest 骨架**（5 分钟）：列出 phase id 和 depends_on，不填内容
3. **写 common.md**（10 分钟）：5–10 条硬约束，从技术栈/非目标/质量底线三个维度各挑几条
4. **写第一个 phase 的两份文档**（10 分钟）：后面的 phase 可以边做边写，不用一次写完
5. **启动**：`ruby scripts/planctl next --format prompt --strict`，按输出开工

第一圈跑通后，后续每个 phase 只需要：`complete` → `handoff --write` → 写下一个 phase 的两份文档 → `next`。

## 11. 七个最容易踩的坑

| # | 坑 | 表现 | 修正 |
|---|----|------|------|
| 1 | phase 切得太粗 | "部分完成"无法处理 | 按"可独立交付"再拆 |
| 2 | execution 写成步骤清单 | AI 照做但越界 | 改成路径白名单 + 禁止项 + 裁决规则 |
| 3 | common.md 混入易变规则 | 流程频繁抖动 | 易变项下放到 phase |
| 4 | 完成判定写主观语言 | AI 自我感觉良好就宣告完成 | 改为可勾选客观项 |
| 5 | 跳过 handoff 刷新 | 压缩恢复失效 | 把 complete + handoff --write 捆绑成一组动作 |
| 6 | 忽视仓库级指令 | AI 直接上手，不跑 resolver | 把指令同时写进 `.github/copilot-instructions.md` / `CLAUDE.md` / `AGENTS.md` 三份 |
| 7 | 让 AI 代写 manifest 和 common | 被 AI 的拟合带偏全局 | 这两份文档必须由人类主导 |

## 12. 适用边界

### 天然适用

- 中大型从 0 到 1 的产品 / 系统
- 框架 / SDK 迁移 / 大版本升级
- 架构替换 / 大型重构
- 长文档工程（技术书、系列报告、课程）
- 合规整改 / 安全加固
- 数据治理 / schema 重建

### 需改造才适用

- **DAG 而非线性**：扩展 planctl 让 `next` 返回"所有依赖已满足的 phase"集合
- **多人协作**：state 加 `assignee`，handoff 加"最近提交人"
- **需要回滚**：给 planctl 加 `revert`，弹出该 phase 及其后所有 phase
- **规模超大**（>15 phase）：引入 epic 分组层

### 不适用

- **探索型研究**：没有预先可定义的 phase 顺序 → 用 ReAct / Plan-and-Solve
- **单次小修复**：上这套反而重 → 直接上手
- **需求高度不稳定**：会频繁重写 phase 文档 → 先进入"规划稳定期"，稳定后再启用

```
  I2 三文件上下文     — 工作窗口只装 common + phase + execution
  I3 完成即事实       — 完成必须落到 state.yaml

七原则：
  P1 状态外部化
  P2 调度与执行分离
  P3 三文件上下文律
  P4 双层合同
  P5 依赖强制校验
  P6 完成即写入
  P7 固定恢复协议

Golden Loop：
  [0] next --strict
  [1] 读 3 份上下文
  [2] 实施（守边界）
  [3] 过交付检查

  → [0]

压缩恢复：
  manifest → handoff → next

切 phase 判据：
  可独立交付 + 可客观验证 + 30–90 分钟

不能违反的一条：
  未写 state 的 phase 不视为完成
```

## 14. 一句话总结

> **把"需要稳定性"的部分外部化为合同，把"需要创造性"的部分留给 AI。**
>
> 合同由人类写一次并长期稳定，AI 在每一圈只面对一个足够小的窗口、按围栏实施、写回事实、刷新锚点，然后自动进入下一圈。
>
> 于是"连续 5 小时"变成了"8 次独立的 40 分钟"，"跨会话续跑"变成了"从文件读而不是从记忆捞"——这就是这套工作流能稳定驱动 AI 长任务的全部秘密。
